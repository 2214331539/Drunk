# 白日微醺项目总结报告

生成时间：2026-04-18

## 1. 项目概况

这是一个基于 React + Vite 的单页 H5 测试应用，主题是“根据当下情绪与 MBTI 推荐微醺酒款”。

项目当前是纯前端实现，没有后端接口、没有路由、没有全局状态库，核心能力都在本地完成：

- 首页展示品牌氛围和开始入口
- 问答页完成 6 道题的选择
- 结果页根据答案生成酒款推荐，并支持滑动查看与图片导出

从代码实现看，它更像一个轻量营销/传播型 H5，而不是复杂业务系统。页面流程短，交互目标明确，适合移动端传播。

## 2. 技术栈与运行方式

### 技术栈

- React 18
- Vite 5
- 原生 CSS
- `html2canvas`，用于把结果卡片导出为图片

### 运行脚本

`package.json` 中定义了以下脚本：

- `npm run dev`：Vite 开发服务，带 `--host`
- `npm start`：Vite 开发服务
- `npm run build`：生产构建
- `npm run preview`：预览构建产物

### 构建验证

我已在当前环境实际执行过 `npm run build`，构建成功。

产物概要：

- `dist/index.html` 约 0.76 kB
- 主 CSS 约 13.20 kB
- 主 JS 约 358.38 kB，gzip 后约 100.76 kB

结论：项目当前可以正常打包上线，没有阻断构建的语法错误。

## 3. 目录结构与职责

### 根目录

- `index.html`：H5 容器页面，设置移动端 viewport、标题、描述和外部字体
- `package.json`：依赖与脚本
- `vite.config.js`：Vite 配置，开发端口为 3000，`server.open = true`
- `README.md`：项目说明，但内容已经落后于当前实现
- `mockdata.json`：空文件，当前未被代码使用

### `src/`

- `main.jsx`：React 挂载入口
- `App.jsx`：应用主控文件，负责问答流程、答案状态和结果计算
- `data/mockdata.json`：核心数据源
- `components/HomePage.jsx`：首页
- `components/QuestionPage.jsx`：问答页
- `components/ResultPage.jsx`：结果页
- `styles/index.css`：全局变量、基础样式
- `styles/App.css`：动画和页面公共样式

## 4. 页面与状态流转

### 4.1 应用状态

`App.jsx` 中只有 4 个核心状态：

- `currentPage`：当前页面，`home / question / result`
- `currentQuestion`：当前题号
- `answers`：用户答案数组
- `result`：最终推荐结果对象

这说明整个应用的控制面集中在单一入口文件中，结构简单，阅读成本低。

### 4.2 页面流程

1. 首页点击“开始”，进入问答页。
2. 问答页逐题作答，共 6 题。
3. 前 5 题是情绪/偏好题，第 6 题是 MBTI。
4. 最后一题提交后调用 `calculateResults` 计算结果。
5. 结果页展示推荐卡片，可左右滑动查看前 6 名推荐。
6. 用户可以重新测试，或导出当前卡片图片。

### 4.3 组件职责

#### HomePage

- 只负责品牌感与视觉引导
- 没有业务逻辑
- 通过 `onStart` 把开始行为交回 `App`

#### QuestionPage

- 接收当前题目数据并渲染选项
- 展示进度条和返回按钮
- 用 `isVisible` 做淡入淡出过渡
- 非 MBTI 题使用纵向列表，MBTI 题切成双列网格

#### ResultPage

- 负责展示 MBTI 标签、当前推荐酒款、风味、酒精度
- 支持触摸滑动和鼠标拖动切换推荐项
- 使用 `html2canvas` 导出卡片 PNG

## 5. 数据结构与推荐算法

### 5.1 题库结构

题库直接写在 `App.jsx` 内，共 6 题：

- 5 道情绪/场景/口味题
- 1 道 MBTI 题

MBTI 题覆盖完整 16 型。

### 5.2 数据源结构

`src/data/mockdata.json` 是项目的核心数据文件，包含：

- 12 种酒款基础资料
- 16 个 MBTI 的人格说明
- 16 组 MBTI 对酒款的匹配数据
- 20 个情绪/口味标签到酒款的加分映射

其中酒款资料包含：

- `id`
- `name`
- `type`
- `flavor`
- `alcohol`
- `image`

当前酒款包括：莫斯卡托起泡酒、长相思、桃红葡萄酒、黑皮诺、梅酒、清酒、起泡酒、桃子气泡酒、金巴利苏打、金汤力、威士忌嗨棒、苹果酒。

### 5.3 评分逻辑

推荐逻辑在 `App.jsx` 的 `calculateResults` 中实现，规则如下：

1. 所有酒先给基础分 `50`
2. 前 5 题答案命中 `moodWineMatches` 时，对对应酒款加分
3. MBTI 命中 `mbtiWineMatches` 时，把 `match - 50` 累加到总分
4. 结果按 `totalScore` 倒序排序
5. 只取前 6 个结果用于展示
6. 展示用匹配度为 `Math.min(99, Math.round(totalScore))`

这个算法的特点：

- 实现简单，易读易改
- 数据驱动程度高，主要通过 JSON 配置控制结果
- 不依赖外部服务，适合静态部署

同时它也有两个明显特征：

- 情绪题只负责“加权”，MBTI 决定最终排序上限更明显
- 展示百分比并不是严格意义上的概率，而是“经过规则加工后的分数显示”

## 6. 视觉与交互实现

### 6.1 实际视觉方向

当前实现并不是 README 中描述的“低饱和柔和粉彩”路线，而是更偏“多巴胺撞色”：

- 主色：`#ff6b6b`
- 辅色：`#4ecdc4`
- 点缀色：`#ffe66d`
- 结果页前三名还使用额外的紫色层级

视觉上更年轻、更活跃，和 README 里的柔沙色、米白色设定已经不是同一套设计语言。

### 6.2 响应式方式

- 移动端为主，`index.html` 限制缩放
- 页面在桌面宽度下会收束到约 420px 的卡片容器内
- 结果页和首页都做了背景渐变、浮动装饰和淡入动画

### 6.3 分享能力

结果页使用 `html2canvas` 将卡片 DOM 渲染为 PNG，再通过临时 `<a>` 标签下载。

这是一个成本低、接入快的方案，适合 H5 场景。

## 7. 当前实现的优点

- 代码量小，核心逻辑集中，阅读和重构门槛低
- 数据与界面分离度还可以，推荐规则主要在 JSON 中定义
- 没有复杂依赖，部署成本低
- 交互闭环完整：首页 -> 答题 -> 推荐 -> 保存图片
- 移动端感知比较强，适合作为传播型活动页雏形

## 8. 已确认的问题与风险

下面这些问题不是推测，而是根据源码和实际构建结果确认过的。

### 8.1 “保存当前卡片”只对第 1 张推荐可靠

位置：

- `src/components/ResultPage.jsx:57-75`
- `src/components/ResultPage.jsx:98-102`

原因：

- `handleShare` 依赖 `cardRef.current`
- 但卡片 DOM 只在 `currentIndex === 0` 时才绑定 `ref`
- 当用户切换到第 2 张及以后时，当前卡片没有挂到 `cardRef`

影响：

- 按钮文案写的是“保存当前卡片”
- 实际行为却偏向“只保存第一张卡片”
- 这是明显的功能与文案不一致

### 8.2 第 4-6 名推荐缺少推荐理由

位置：

- `src/App.jsx:112-133`
- `src/components/ResultPage.jsx:118`

原因：

- 每个 MBTI 只配置了 3 条 `reason`
- 但结果页固定展示前 6 名
- `matchReason` 只有命中 MBTI 三个主推荐时才会被写入

我用一组样例答案验证后，前 3 张卡片有理由，第 4-6 张卡片会出现 `hasReason: false`。

影响：

- 结果页后半段内容信息密度明显下降
- 会出现空白理由区，影响成品感和分享质量

### 8.3 README 已与实际代码明显漂移

位置：

- `README.md:22-38`
- `README.md:66-101`
- `src/styles/index.css:8-30`
- `src/App.jsx:9-82`

主要不一致点：

- README 仍写 `App.js`、`index.js`，实际是 `App.jsx`、`main.jsx`
- README 写了 `public/index.html`，实际入口在根目录 `index.html`
- README 说问答页是 5 道题，实际是 6 题
- README 中“修改结果逻辑”写的是 `getResult`，实际函数名是 `calculateResults`
- README 的视觉规范是低饱和柔和配色，实际代码是高饱和撞色风格

影响：

- 新接手的人会先被错误文档误导
- 二次开发时容易改错位置

### 8.4 存在无用或易混淆资产

位置：

- 根目录 `mockdata.json`

现状：

- 该文件大小为 0
- 当前应用实际使用的是 `src/data/mockdata.json`

影响：

- 容易让维护者误判真实数据源
- 后续如果有人误改根目录文件，实际上不会生效

### 8.5 外部字体依赖有部署风险

位置：

- `index.html:9-11`

现状：

- 页面直接引用 Google Fonts 的 `Noto Serif SC`

风险：

- 在受限网络环境下字体可能加载失败
- H5 的首屏字体表现会受外部网络质量影响

### 8.6 缺少自动化质量保障

现状：

- 项目中没有测试文件
- 没有 ESLint、Prettier、TypeScript 或类型检查链路

影响：

- 目前依赖人工回归
- 小项目阶段可接受，但一旦继续迭代，回归风险会快速上升

### 8.7 轻微代码清理项

位置：

- `src/App.jsx:1` 中的 `useMemo` 未使用
- `src/components/ResultPage.jsx:10` 的 `containerRef` 没有实际参与逻辑

影响：

- 对运行影响不大
- 但会增加“代码似乎还有未完成意图”的维护噪音

## 9. 综合判断

这个项目目前处于“可运行、可展示、适合继续打磨”的状态。

如果把它看成一个活动型 H5 原型，它已经具备：

- 完整路径
- 明确主题
- 数据驱动推荐
- 基本的传播分享能力

但如果要作为正式交付版本，至少应该优先处理以下三件事：

1. 修正结果页导出逻辑，让“保存当前卡片”真正保存当前卡片
2. 处理第 4-6 张推荐没有理由的问题，缩到前 3 张，或为后 3 张补文案
3. 更新 README，确保文档和当前实现一致

## 10. 建议的后续迭代顺序

### 第一优先级

- 修复结果卡片导出 bug
- 决定结果页到底展示 3 张还是 6 张
- 去掉空的根目录 `mockdata.json`

### 第二优先级

- 更新 README 和项目结构说明
- 增加基础 lint
- 给推荐结果增加兜底文案，避免空白区

### 第三优先级

- 将题库与题目配置也抽离到独立 JSON
- 为移动端滑动行为改成统一 Pointer Events
- 评估是否改成本地字体或自托管字体

## 11. 一句话结论

这是一个结构清晰、定位明确、适合静态部署的 React H5 项目；核心流程已经跑通，但结果页导出与推荐内容完整性还存在明显产品级缺口，文档也需要尽快补齐。
